**Release Final del Proyecto**  
**Proyecto Integrador de Ciencias de Datos: Presentación Final**

**Objetivo:**

Este entregable tiene como objetivo consolidar el proyecto como un **producto reproducible, documentado y comunicable**, evaluando la capacidad del equipo para presentar y defender profesionalmente la solución desarrollada.

Se espera que el sistema pueda ser ejecutado por terceros a partir de instrucciones claras, y que el equipo pueda justificar las decisiones tomadas, reflexionar sobre el proceso y comunicar el valor del proyecto.

El foco está en la **calidad de la entrega final**, tanto técnica como comunicacional.

**Consigna:**  
Cada equipo deberá:

1. Entregar una versión final del sistema que pueda ser **ejecutada de manera reproducible** (por ejemplo, mediante Docker Compose o mecanismo equivalente).  
2. Presentar una **documentación completa**, tanto técnica como orientada al usuario.  
3. Realizar una **presentación final de 15-20 minutos**, con demo incluida.  
4. Elaborar un **post-mortem técnico**, reflexionando sobre el desarrollo del proyecto.

**Rúbrica de evaluación:**  
La presentación debe cubrir los siguientes aspectos, cada uno con un peso del 25% en la evaluación:

| Aspecto (% evaluación) | Contenido |
| ----- | ----- |
| Reproducibilidad y entrega técnica (25%) | El sistema puede ser ejecutado por terceros siguiendo instrucciones claras. Uso de Docker Compose o mecanismo equivalente para levantar el sistema. Configuración coherente del entorno (dependencias, servicios, variables). El sistema funciona correctamente en un entorno limpio. |
| Documentación del proyecto (25%) | README completo (instalación, ejecución, uso del sistema). Documentación técnica (arquitectura, componentes, flujo de datos). Documentación de usuario (cómo interactuar con el sistema). Organización clara del repositorio. |
| Presentación y comunicación (25%) | Presentación clara, estructurada y dentro del tiempo. Explicación del problema, solución y valor del proyecto. Demo breve, controlada y alineada con el discurso. Capacidad de comunicar a una audiencia mixta. |
| Post-mortem técnico (25%) | Identificación de decisiones clave durante el desarrollo. Análisis de dificultades encontradas y cómo fueron abordadas. Reflexión sobre limitaciones del sistema. Propuestas concretas de mejora o trabajo futuro. Evidencia de aprendizaje a lo largo del proyecto. |

**Checklist pre-vuelo (verificar antes de la demo):**

- El sistema se levanta en una máquina nueva sin intervención manual compleja.  
- El docker-compose (o equivalente) funciona correctamente.  
- El README permite a un tercero ejecutar el proyecto.  
- La demo está preparada y controlada.  
- La presentación tiene un hilo claro.  
- El post-mortem es honesto (no solo “todo salió bien”).

**Formato de entrega:**

- Repositorio GitHub final.  
- Configuración de ejecución reproducible (Docker Compose o equivalente).  
- README completo.  
- Documentación técnica y de usuario.  
- Presentación (slides).  
- Documento de post-mortem técnico.

**Formato de presentación:** Exposición oral de 15-20 minutos por grupo, con demo incluida.

**Evaluación:** Se valorará especialmente:

- Se valorará especialmente:  
- la capacidad de entregar un sistema reproducible y ejecutable por terceros;  
- la calidad y claridad de la documentación;  
- la solidez en la comunicación del proyecto;  
- la capacidad de análisis crítico en el post-mortem;  
- la coherencia global de la solución desarrollada.
